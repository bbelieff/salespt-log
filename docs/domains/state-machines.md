# 상태 전이 다이어그램

## 1. Meeting 상태 전이 (ADR-0003 업데이트)

```mermaid
stateDiagram-v2
    [*] --> 예약 : POST /api/meeting (업체관리 탭 추가)
    
    예약 --> 완료 : PATCH /api/meeting/:id (상태=완료)
    예약 --> 취소 : PATCH /api/meeting/:id (상태=취소)
    
    완료 --> 계약성사 : PATCH /api/meeting/:id (계약여부=true, 수임비 입력)
    완료 --> 계약실패 : PATCH /api/meeting/:id (계약여부=false)
    
    계약성사 --> [*] : 수임비+계약비고 입력 완료
    계약실패 --> [*] : 미팅 종료 처리
    취소 --> [*] : 취소 사유 입력 완료
    
    note right of 예약 : J열=예약, K열=false\n업체관리 탭 A~M열
    note right of 완료 : J열=완료\n영업관리 탭 수식 자동 업데이트
    note right of 계약성사 : K열=true, L열=수임비\nM열=계약비고 입력
```

### Meeting 필드별 상태 변화
- **예약 단계**: A~I열 입력, J=예약, K=false, L=0, M=빈값
- **완료 단계**: J=완료 (K, L, M 유지)
- **계약성사**: J=완료, K=true, L=수임비, M=계약비고
- **계약실패**: J=완료, K=false (L, M 변경 없음)
- **취소**: J=취소, 나머지 필드 유지

## 2. Payment 상태 전이 (ADR-0003 업데이트)

```mermaid
stateDiagram-v2
    [*] --> 입력완료 : POST /api/payment (수납관리 탭 추가)
    
    입력완료 --> 수정 : 사용자가 기존 데이터 수정 요청
    수정 --> 입력완료 : 수정 내용 저장
    
    입력완료 --> [*] : 일별 수납 기록 확정
    
    note right of 입력완료 : 수납관리 탭 A~F열\n일별 단위로 기록
    note right of 수정 : 같은 날짜 데이터\n수정/업데이트
```

### Payment 단순 상태 모델 (ADR-0003)
Payment는 Meeting과 달리 **단순 입력 모델**을 따름:
- **독립성**: Meeting과 별개의 워크플로우
- **일별 기록**: 1행 = 1일 수납 기록
- **상태 없음**: 복잡한 승인/수납 상태 전이 대신 단순 CRUD
- **필드**: A(id), B(수납날짜), C(승인건수), D(수납건수), E(수납금액), F(기관비고)

## 3. DailyEntry 저장 상태 전이

```mermaid
stateDiagram-v2
    [*] --> draft : 사용자가 데이터 입력 시작
    
    draft --> 검증중 : 저장 버튼 클릭 (사용자 액션)
    
    검증중 --> 검증실패 : 필수값 누락/형식 오류 (시스템 검증)
    검증중 --> 저장중 : 검증 통과 (시스템)
    
    검증실패 --> draft : 사용자가 오류 수정
    
    저장중 --> saved : Google Sheets 저장 성공 (시스템)
    저장중 --> 저장실패 : 네트워크/권한 오류 (시스템)
    
    저장실패 --> draft : 사용자가 재시도
    saved --> [*] : 저장 완료
    
    note right of draft : 프론트엔드 로컬 상태\n실시간 입력 반영
    note right of 검증중 : 클라이언트 측 유효성 검사\n필수값, 숫자 형식 등
    note right of 저장중 : API 호출 진행\nGoogle Sheets 업데이트
```

```mermaid
stateDiagram-v2

    임시저장 --> 확정 : 주문 확정 (사용자 액션)

    확정 --> 진행중 : 시스템에서 처리 시작
    확정 --> 취소 : 사용자 취소 요청
    
    진행중 --> 완료 : DB 공급 완료 (시스템/관리자)
    진행중 --> 취소 : 공급 중단 (시스템/관리자)

    취소 --> [*] : 취소 처리 완료
    
    note right of 입력 : DB관리 탭에서\n수량/채널 입력
    note right of 확정 : 주문 확정 시\n재고에서 차감
    note right of 완료 : 실제 DB 전달 완료\n효율 지표에 반영
```

## 상태 전이 트리거 요약

### 사용자 액션 트리거 (ADR-0003 업데이트)
- **Meeting**: 미팅 예약 생성(POST), 상태 변경(PATCH), 계약 정보 입력
- **Payment**: 수납 데이터 입력(POST), 기존 데이터 수정
- **DailyEntry**: 컨택 데이터 입력, 특이사항 입력, 저장 요청
- **DBOrder**: 주문 입력, 확정 처리, 취소 요청

### 시스템 이벤트 트리거 (ADR-0003 업데이트)
- **Meeting**: 업체관리 탭 → 영업관리 탭 시트 수식 자동 집계
- **Payment**: 수납관리 탭 → 영업관리 탭 시트 수식 자동 집계
- **DailyEntry**: 클라이언트 검증, API 응답, 시트 저장 완료
- **DBOrder**: 재고 관리 시스템, 관리자 처리

### 상태 전이 제약 조건 (ADR-0003 업데이트)
1. **Meeting**: 
   - 예약 → 완료: 미팅날짜/시간 검증
   - 완료 → 계약성사: 수임비 > 0, 계약비고 필수
   - 같은 업체명+미팅날짜 중복 허용 (업체 마스터 없음)
2. **Payment**: 
   - 승인건수 ≥ 수납건수 (승인 후 입금)
   - 수납금액 > 0
   - 하루 1개 기록만 허용 (수정으로 처리)
3. **DailyEntry**: 
   - 필수 필드: date, channels (4개 채널 전체)
   - 채널별 생산 ≥ 유입 ≥ 컨택진행 ≥ 컨택성공 (퍼널 순서)
4. **DBOrder**: 재고 수량 확인 후 확정 처리 (기존 유지)

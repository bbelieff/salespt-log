# 향후 확장 기능 로드맵

## 확장 원칙

### YAGNI (You Aren't Gonna Need It)
- MVP에서 의도적으로 제외된 기능들을 체계적으로 관리
- "언젠가 필요할 수도"가 아닌 **명확한 트리거**가 있을 때만 구현
- 각 기능별로 "왜 MVP에서 제외했는가" 명시

### 우선순위 기준
- **P2 (단기)**: MVP 출시 후 3개월 내 구현 검토
- **P3 (중기)**: 6개월~1년 후 사업 확장 시점
- **P4 (장기)**: 1년 후 플랫폼 전환 시점

## P2: 단기 확장 기능 (3개월 내)

### 1. Google Calendar 동기화
**기능**: 미팅 예약 시 Google Calendar에 자동 등록

**MVP 제외 이유**:
- Google Calendar API 추가 권한 필요 (OAuth scope 확장)
- 실패 처리 복잡성 (Calendar 실패해도 미팅 저장은 성공해야 함)
- 핵심 가치(기록)와 무관한 편의 기능

**구현 트리거**:
- 수강생 5명 이상이 Calendar 연동을 요청할 때
- 미팅 관리 부담으로 인한 앱 사용 포기 사례 발생 시

**구현 방향**:
```typescript
// 미팅 저장 로직 확장
async function createMeetingWithCalendar(meetingData) {
  // 1. 시트 저장 (핵심 기능, 절대 실패 안함)
  const meetingId = await saveMeetingToSheet(meetingData);
  
  // 2. Calendar 동기화 (부가 기능, 실패해도 OK)
  try {
    const calendarEventId = await calendar.createEvent(meetingData);
    await updateMeetingCalendarId(meetingId, calendarEventId);
  } catch (error) {
    // 시트는 저장됨, Calendar만 실패
    showWarningToast("미팅 저장 완료. 캘린더 동기화는 실패했습니다.");
  }
}
```

**예상 공수**: 2주 (Calendar API 연동 + 에러 처리)

### 2. 푸시 알림 시스템
**기능**: 일일 기록 누락 시 알림, 목표 달성 시 축하 알림

**MVP 제외 이유**:
- PWA Service Worker 복잡성
- 알림 권한 요청이 사용자 경험 방해 가능성
- 기록 습관이 안정되기 전에는 스팸성 알림될 위험

**구현 트리거**:
- 수강생들의 기록 일관성이 70% 이하로 떨어질 때
- "깜빡했는데 알림이 있었으면 좋겠다" 피드백 3건 이상

**구현 방향**:
- 매일 오후 11시 기록 누락 체크
- 스트릭 3일 달성, 주간 목표 달성 등 긍정적 알림 위주
- 사용자가 알림 빈도/시간을 설정 가능

**예상 공수**: 1주 (Service Worker + 스케줄링)

### 3. 오프라인 모드 개선
**기능**: 완전 오프라인에서도 입력 가능, 재연결 시 자동 동기화

**MVP 제외 이유**:
- IndexedDB, 동기화 로직 복잡성
- 충돌 해결 UX 설계 필요
- WiFi 환경에서 사용하는 수강생이 대부분

**구현 트리거**:
- 지하철/버스에서 사용하려는 수강생 증가
- 네트워크 문제로 인한 데이터 손실 사고 발생

**구현 방향**:
- 현재 localStorage 기반을 IndexedDB로 확장
- 큐 기반 동기화 + 충돌 감지
- "동기화 대기 중" UI 상태 추가

**예상 공수**: 3주 (데이터 레이어 재설계)

## P3: 중기 확장 기능 (6개월~1년)

### 1. 수료생 구독 서비스
**기능**: 8주 수료 후 월 구독으로 지속 사용 가능

**MVP 제외 이유**:
- 결제 시스템 (PG 연동) 복잡성
- 수료생 요구사항이 현재 수강생과 다를 가능성
- 사업 모델 검증 필요

**구현 트리거**:
- 5기 이상 수료생 중 30% 이상이 지속 사용 의향 표시
- 세일즈PT에서 수료생 대상 구독 서비스 론칭 결정

**기능 확장**:
- 무제한 기록 기간 (8주 제한 해제)
- 월간/분기별 리포트
- 수료생 전용 커뮤니티 기능 (선택사항)

**예상 공수**: 4주 (결제 + 권한 체계 + UI 확장)

---

### 2. 트레이너 대시보드
**기능**: 트레이너가 수강생들의 진도를 한눈에 볼 수 있는 관리 화면

**MVP 제외 이유**:
- 현재는 수강생 1인칭 관점에 집중
- 개인정보보호 이슈 (수강생 동의 필요)
- 트레이너 요구사항 정확히 파악 안됨

**구현 트리거**:
- 트레이너로부터 수강생 관리 도구 요청 시
- 수강생 규모가 한 기수 20명 이상으로 확대 시

**기능 범위**:
- 수강생별 기록 일관성 현황
- 미팅/계약 성과 집계 (익명화)
- 뒤처지는 수강생 얼리 워닝 (선택사항)

**예상 공수**: 6주 (권한 체계 + 새 UI + 개인정보처리)

---

### 3. 월간/분기 리포트
**기능**: 장기 트렌드 분석, PDF 내보내기, 포트폴리오 생성

**MVP 제외 이유**:
- 8주 수강 기간에는 월간 트렌드 의미 제한적
- PDF 생성 라이브러리 의존성 추가
- 리포트 형식에 대한 명확한 요구사항 없음

**구현 트리거**:
- 수료 후에도 계속 사용하는 수강생 20명 이상 확보
- "내 영업 포트폴리오를 만들고 싶다" 요청 증가

**기능 범위**:
- 월간/분기별 성과 그래프
- PDF 다운로드 + 인쇄 최적화
- 개인 인사이트 텍스트 생성 (AI 활용 가능)

**예상 공수**: 3주 (차트 라이브러리 + PDF)

## P4: 장기 확장 기능 (1년 후 플랫폼 전환)

### 1. 일반 영업인용 SaaS 버전
**기능**: 세일즈PT와 무관한 일반 영업직군도 사용 가능한 버전

**MVP 제외 이유**:
- 완전히 다른 타겟 사용자 (요구사항 불명)
- 수강 기간 제약 없는 버전으로 아키텍처 변경 필요
- 사업 모델과 마케팅 전략 재검토 필요

**구현 트리거**:
- 세일즈PT 외부에서 앱 사용 문의 월 10건 이상
- 경쟁 제품 분석 후 차별화 포인트 확보
- 별도 팀/예산 확보 결정

**기능 확장**:
- 업종별 커스터마이징 (부동산, 보험, IT 등)
- 팀/조직 기능, 상급자 리포팅
- 유료 플랜별 기능 차등

**예상 공수**: 6개월 (새 제품급 재개발)

---

### 2. 조직/팀 기능
**기능**: 여러 영업사원이 속한 회사/팀에서 사용하는 협업 도구

**MVP 제외 이유**:
- 현재는 개인 기록 도구로 설계됨
- 조직 계층, 권한, 비교/경쟁 요소 복잡성
- B2B 영업 vs B2C 개인 고객 차이

**구현 트리거**:
- 여러 수료생이 같은 회사에서 근무하며 팀 단위 사용 요청
- 영업 관리자들의 팀 관리 도구 문의 증가

**기능 범위**:
- 팀별 목표 설정 및 달성률 비교
- 팀원 간 노하우 공유 (선택사항)
- 관리자 대시보드, 개인정보 보호 설정

**예상 공수**: 4개월 (권한 시스템 + 협업 UI)

---

### 3. AI 분석 및 추천
**기능**: 영업 패턴 분석 후 개선 포인트 자동 제안

**MVP 제외 이유**:
- AI 모델 개발 및 학습 데이터 부족
- 영업 도메인 전문성 필요
- 잘못된 추천이 오히려 해가 될 위험

**구현 트리거**:
- 충분한 데이터 확보 (최소 100명×8주 = 800주차 데이터)
- 영업 전문가와의 협업 체계 구축
- AI 기술 내재화 또는 전문 파트너 확보

**기능 범위**:
- "이 채널의 전환율이 떨어지고 있습니다" 패턴 감지
- "비슷한 수강생들은 이런 방법을 썼어요" 벤치마킹
- 개인 맞춤형 목표 설정 제안

**예상 공수**: 6개월 (데이터 파이프라인 + 모델 + UI)

## 확장 기능 관리 원칙

### 1. 트리거 기반 개발
- 각 기능은 명확한 **트리거 조건**이 충족될 때만 착수
- 사용자 요청, 데이터 기반 근거, 사업적 필요성 중 최소 2개 이상

### 2. 핵심 가치 보존
- 모든 확장 기능은 "8주 수강생의 영업 기록"이라는 핵심 가치를 해치지 않아야 함
- 복잡성이 사용자 경험을 해치면 과감히 포기

### 3. 점진적 확장
- 한 번에 여러 P2 기능을 동시 개발하지 않음
- 하나 완료 → 사용자 피드백 → 다음 기능 착수

### 4. MVP 아키텍처 호환성
- 기존 Google Sheets 기반 아키텍처를 최대한 활용
- 새 기능으로 인한 기존 기능 리그레션 방지
- 필요시 점진적 마이그레이션 (빅뱅 리팩터링 금지)

## 의사결정 프로세스

### 기능 요청 접수
1. **사용자 피드백**: GitHub Issues, 이메일, 수강생 면담
2. **데이터 기반 판단**: 사용 패턴 분석, A/B 테스트 결과
3. **사업적 임팩트**: ROI 계산, 리소스 투입 대비 효과

### 우선순위 재조정
- **매달 1회**: P2 기능들의 트리거 조건 재검토
- **분기별 1회**: P3 기능 로드맵 업데이트
- **연간 1회**: P4 기능 전략 재수립

### 기능 중단/보류 기준
- 개발 완료 후 사용률 20% 미만 시 deprecation 검토
- 유지보수 비용이 가치보다 높아질 때 단계적 제거
- 더 나은 대안 기능 출현 시 교체

이러한 체계적인 확장 계획을 통해 MVP의 단순함을 유지하면서도 미래 성장 가능성을 열어둘 수 있습니다.

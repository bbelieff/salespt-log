# ADR-0003: 업체관리·수납관리 탭 분리

## 📄 이 문서는 무엇인가요?

영업관리 탭의 미팅/계약 기록 시 DB 무결성 문제를 해결하기 위해 **업체관리 탭**과 **수납관리 탭**을 신설하는 아키텍처 결정 기록입니다.

**상태**: `approved` | **날짜**: 2026-04-21 | **결정자**: bbelieff

---

## Context

### 현재 문제점

영업관리 탭의 I~P 컬럼(미팅/계약 기록)을 웹에서 직접 기록할 때, Google Sheets의 셀 내 줄바꿈(`CHAR(10)`)으로 인한 데이터 파싱 문제:

- 여러 미팅을 한 셀에 줄바꿈으로 구분해 저장
- 웹에서 개별 미팅 편집/삭제가 불가능
- 파싱 오류 시 전체 데이터 손실 위험
- 미팅과 수납이 같은 컬럼 구조에 혼재

### 비즈니스 요구사항

1. **미팅 기록의 정확성**: 1미팅 = 1행으로 개별 관리 필요
2. **예약일 vs 미팅날짜 분리**: 
   - 예약을 잡은 날(생산성 지표)
   - 실제 미팅 진행 날(컨택 지표)
3. **수납 독립성**: 미팅과 별개로 수납 내역 관리
4. **기존 대시보드 호환**: 현재 수식 기반 집계 유지

---

## Decision

### 1. 새로운 탭 구조

#### A. 업체관리 탭 (신설)
- **목적**: 1미팅 = 1행으로 정규화된 미팅 데이터 저장
- **역할**: 웹의 직접 쓰기 대상, 영업관리 탭은 수식으로만 참조

#### B. 수납관리 탭 (신설)  
- **목적**: 미팅과 독립적인 수납 기록 관리
- **역할**: 승인/수납 현황을 별도 추적

#### C. 영업관리 탭 (기존, 역할 변경)
- **기존**: 직접 데이터 입력 + 수식 집계
- **변경**: 수식 집계만 (I~L, N~T 컬럼)
- **유지**: 직접 입력 (E~H, M 컬럼 - 생산활동)

### 2. 데이터 흐름

```
웹 입력 → 업체관리/수납관리 탭 → (수식) → 영업관리 탭 → (수식) → 대시보드
```

### 3. 핵심 설계 원칙

1. **예약일 vs 미팅날짜 분리**:
   - I열: 예약일 기준 집계 (생산성)
   - J열: 미팅날짜 기준 집계 (컨택)
2. **표시문자열 2종류**:
   - N열(상세): 날짜 포함 `4/25 13:00 ○○치킨 방이동`
   - O열(요약): 날짜 제외 `13:00 ○○치킨 방이동`
3. **MVP 범위**: 업체 마스터 없이 매번 새 행 생성
4. **수식 기반 집계**: 웹은 원본 데이터만 생성, 집계는 Sheets가 담당

---

## Consequences

### Positive

- ✅ **데이터 무결성**: 1미팅 = 1행으로 파싱 오류 방지
- ✅ **개별 편집**: 미팅별 상태/결과 수정 가능
- ✅ **명확한 지표**: 예약일/미팅날짜 분리로 정확한 생산성 측정
- ✅ **확장성**: 향후 업체 마스터, 히스토리 추가 용이
- ✅ **기존 호환**: 대시보드 수식 구조 유지

### Negative

- ❌ **시트 복잡도 증가**: 탭 2개 추가 (5개 → 7개)
- ❌ **수식 의존도**: 업체관리 → 영업관리 수식 오류 시 전체 영향
- ❌ **초기 마이그레이션**: 기존 데이터 재구조화 필요

### Neutral

- 🔄 **API 변경**: 기존 영업관리 쓰기 API → 업체관리 API로 전환
- 🔄 **개발 복잡도**: 탭간 관계 관리 로직 추가

---

## Alternatives Considered

### A. 현재 구조 유지 + 파싱 로직 개선
- **장점**: 시트 구조 변경 없음
- **단점**: 근본적 파싱 문제 해결 불가, 개별 편집 여전히 불가능
- **결론**: 기술적 한계로 인해 거부

### B. JSON 컬럼 도입
- **장점**: 구조화된 데이터 저장
- **단점**: Sheets 수식에서 JSON 파싱 복잡, 가독성 저하
- **결론**: Sheets 철학(가독성)에 맞지 않아 거부

### C. 외부 DB 도입
- **장점**: 정규화된 관계형 모델
- **단점**: MVP 스코프 위반 (SSOT = Sheets 원칙)
- **결론**: 아키텍처 제약 위반으로 거부

---

## Implementation Notes

### 마이그레이션 전략
1. 새 탭 구조 생성 (업체관리, 수납관리)
2. 기존 영업관리 데이터를 업체관리로 이관
3. 영업관리 I~L, N~T를 수식으로 교체
4. API 엔드포인트 업데이트
5. 프론트엔드 폼 수정

### 검증 기준
- [ ] 업체관리 N/O 컬럼의 표시문자열이 정확히 생성
- [ ] 영업관리 I열(예약일 기준) ≠ J열(미팅날짜 기준) 확인
- [ ] 수납관리 독립 동작 확인
- [ ] 기존 대시보드 수식 정상 작동
